# ADR-016: Config and Cache File Separation

**Status**: Accepted  
**Date**: 2026-06-23  
**Deciders**: Paul-Elian Tabarant

## Context

The `ripe` CLI needs to persist two types of data locally:

1. **Project registration data**: `serverUrl` and `projectId` — should be shared across all team
   members working on the same project
2. **Skill ID mappings**: `(skill_name → skill_id)` cache — developer-local, can be regenerated
   by re-registering skills

Initially, the design proposed a single gitignored cache file. However, this meant the first
developer to run `ripe init` would register the project and get a `project_id`, but other
developers would have no way to discover it (the cache is gitignored). Each developer would create
a new project registration, fragmenting telemetry data across multiple project IDs.

## Decision

Split into two files with different git tracking:

- **`.ripe/config.json`** (committed to git) — stores `serverUrl` and `projectId` (camelCase)
- **`.ripe/cache.json`** (gitignored) — stores skill ID mappings `(skill_name → skill_id)`

The first developer to run `ripe init` commits the config file. Subsequent developers inherit the
`projectId` from git and use the same project registration.

## Rationale

**Why commit `projectId`:**

- All team members need to report telemetry to the same project
- Re-registering per developer would create N projects for N developers (data fragmentation)
- The `projectId` is not sensitive (it's a server-assigned identifier, not credentials)

**Why gitignore skill cache:**

- Skill ID mappings can be regenerated by re-registering skills with the server
- Each developer may have different local skills installed (e.g., experimental/personal skills)
- Losing the cache is non-destructive — next `ripe sync` recreates it

**Why camelCase in config:**

- User explicitly requested camelCase for config file properties
- Aligns with JavaScript/TypeScript convention for configuration objects

## Alternatives Considered

- **A. Single gitignored cache for everything** — first developer gets a `project_id`, others
  create duplicates; telemetry data fragments across projects; ruled out.
- **B. Single committed file for everything** — skill cache would pollute git history with frequent
  churn; developers with different local skills would have merge conflicts; ruled out.
- **C. Environment variable for `project_id`** — requires manual setup on every developer machine;
  no single source of truth in the repo; ruled out.

## Consequences

**Positive**:

- ✅ All team members automatically share the same `projectId` via git
- ✅ First-time setup is simple: `ripe init <url>`, commit `.ripe/config.json`, done
- ✅ Skill cache regenerates automatically if lost (non-destructive)
- ✅ Clear separation: committed config for team-shared state, gitignored cache for local state

**Risks/Trade-offs**:

- ⚠️ If config is accidentally gitignored, developers would create duplicate projects (mitigated
  by: `.ripe/config.json` explicitly listed in git, not in `.gitignore`)
- ⚠️ If a developer loses their skill cache, the next `ripe sync` makes N HTTP requests to
  re-register (acceptable — happens rarely, non-blocking failure mode)
