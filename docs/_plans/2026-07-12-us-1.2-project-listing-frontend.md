# US-1.2 — Project Listing: Frontend — Implementation Plan

**Spec**: `docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md`
**Status**: Approved — ready for implementation

## Context

Backend (`GET /api/projects`) is already done (`api/src/endpoints/projectEndpoints.ts`).
`web/` does not exist yet — deferred to Slice 2 by the walking-skeleton plan
(`docs/_plans/2026-06-27-us-1.1-walking-skeleton.md`). This plan bootstraps `web/` and builds
the listing feature on top, plus wires the API to serve the built SPA.

Architecture: ADR-008 (frontend layering: pages / atomic components / hooks / services),
ADR-009 (testing strategy: MSW only at the real network boundary, everything above the service
layer tested with the service mocked via DI).

## Decisions made during planning

- **Styling**: Tailwind v4 (`@tailwindcss/vite` plugin), no component library (no shadcn).
- **Atoms**: only what the feature strictly needs — `LoadingSpinner`, `ErrorMessage`,
  `EmptyState`. No generic `Button`/`Select` atom; the dropdown is a plain `<select>` inside the
  `ProjectSelector` molecule.
- **Hydration vs. wiring**: `main.tsx` mounts `<App/>` (React root). `App` renders
  `DashboardPage`. Per ADR-008, the *page* — not `main.tsx`/`App` — is what wires `useProjects`
  to the `ProjectListing` organism and owns `selectedProjectId` state.
- **Static serving**: in scope. `api/app.ts` serves `web/dist` via `@fastify/static`, with SPA
  fallback to `index.html` for non-`/api` GET requests. `buildApp` gets an optional `staticRoot`
  param (DI, same pattern as existing options) so tests can point at a fixture dir instead of a
  real build.
- **Browser persistence**: explicitly out of scope per spec ("no browser persistence — resets on
  refresh") — nothing to build.
- **E2E (Playwright)**: out of scope for this slice — no full browser journey test yet.

## Files

**Bootstrap** (`web/`, commit separately once green):

- `package.json`, `vite.config.ts` (with `@tailwindcss/vite`), `vitest.config.ts`, `tsconfig.json`,
  `index.html`, `src/index.css` (`@import "tailwindcss";`)
- deps: `react`, `react-dom`, `vite`, `@vitejs/plugin-react`, `tailwindcss`, `@tailwindcss/vite`,
  `typescript`, `vitest`, `jsdom`, `@testing-library/react`, `@testing-library/jest-dom`,
  `@testing-library/user-event`, `msw`, `@biomejs/biome`
- scripts mirror `api/`: `dev`, `build` (`vite build`), `lint` (`biome ci`), `typecheck`
  (`tsc --noEmit`), `test` (`vitest run`), `ci:checks`
- add `web` to `pnpm-workspace.yaml`

**Feature** (`web/src/`):

- `services/projectService.ts` — `fetchProjects(): Promise<Project[]>`
- `hooks/useProjects.ts` — `useProjects(fetchProjectsFn = fetchProjects)` →
  `{status: 'loading'|'error'|'success', projects}`
- `components/atoms/{LoadingSpinner,ErrorMessage,EmptyState}.tsx`
- `components/molecules/ProjectSelector.tsx` — `<select>`, no default selection, `onSelect(id)`
- `components/organisms/ProjectListing.tsx` — presentational, state → view
- `pages/DashboardPage.tsx` — wires hook → organism, owns `selectedProjectId`, accepts optional
  `fetchProjectsFn` for test injection
- `App.tsx`, `main.tsx`

**Tests** (`web/tests/`):

- `services/projectService.test.ts` — MSW, real fetch, success/error
- `components/molecules/ProjectSelector.test.tsx` — pure render/selection, no network
- `pages/DashboardPage.test.tsx` — the 5 Gherkin scenarios end-to-end, service mocked via DI

**API** (independent subtask, different package):

- `api/package.json` — add `@fastify/static`
- `api/src/app.ts` — register static plugin, SPA fallback, optional `staticRoot` param
- `api/tests/` — new test using a fixture static dir (not a real `web/dist` build)

## Scenario coverage

| Scenario | Covered by |
| --- | --- |
| Loading — spinner only | `DashboardPage.test.tsx` |
| Projects available — dropdown, no default selection | `DashboardPage.test.tsx` + `ProjectSelector.test.tsx` |
| Selecting a project | `DashboardPage.test.tsx` + `ProjectSelector.test.tsx` |
| No projects — empty state | `DashboardPage.test.tsx` |
| Request fails — error state | `DashboardPage.test.tsx` |

## Not in scope

- Browser persistence of selection.
- Playwright e2e journey.
- Project deletion/editing, filtering/searching (per spec).
- Activity view (US-3.1).

## Execution notes

Two independent implementers (disjoint packages, no shared files):

1. **Web**: bootstrap `web/` first (must be green before feature code, since tests need the
   harness working), then TDD the feature bottom-up — service → hook → atoms → molecule →
   organism → page, per unit: red → simplest implementation → green → refactor.
2. **API**: add `@fastify/static`, wire SPA fallback, DI `staticRoot` — TDD via a fixture static
   dir, no dependency on the web build existing.

Merge both, run `pnpm --filter web ci:checks` and `pnpm --filter api ci:checks`, then continue
dev-iteration from step 4 (merge) onward.
