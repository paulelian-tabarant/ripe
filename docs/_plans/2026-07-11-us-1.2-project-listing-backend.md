# US-1.2 — Project Listing: Backend (`GET /api/projects`) — Implementation Plan

**Status**: Approved, not yet implemented.

## Context

`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md` needs a
`GET /api/projects` endpoint so the frontend can list registered projects for
selection. Alongside it, the api's business-logic layer switches from service
classes (one class, many methods) to **use-case classes** (one class per use
case, `run()` as the single public method, repository still constructor
injected). This touches the existing `registerProject` use case (currently
`ProjectService`) as well as the new `listProjects` one, so both land here,
along with `api/STANDARDS.md` / `api/CLAUDE.md` updates describing the new
pattern.

Ordering of the list is **not** guaranteed by the backend — sorting (if any)
is a frontend concern.

## Architecture change: services → use-cases

- New directory `api/src/use-cases/` replaces `api/src/services/`.
- One class per use case, constructor takes the repository (or repositories)
  it needs, single public method named `run()`.
- `api/src/repositories/ProjectRepository.ts` — unchanged in shape, just gets
  a new `findAll()` method.

## Files

1. **`api/src/repositories/ProjectRepository.ts`**
   - Add `findAll(): Project[]` backed by a prepared statement:
     `SELECT id, name FROM projects` (no `ORDER BY`).

2. **`api/src/use-cases/RegisterProject.ts`** (moved from
   `api/src/services/ProjectService.ts`, renamed class `ProjectService` →
   `RegisterProject`, method `registerProject(name)` → `run(name)`). Keeps
   `RegisterProjectResult` type as-is.

3. **`api/src/use-cases/ListProjects.ts`** (new)
   - `class ListProjects { constructor(private readonly repository: ProjectRepository) {}`
     `run(): Project[] { return this.repository.findAll(); } }`
   - Reuses the repository's `Project` type (`{ id, name }`) as-is — API
     response shape is `{ id, name }[]`, so no field renaming/mapping needed
     between repository and route.

4. **`api/src/routes/projectRoutes.ts`**
   - `ProjectRouteOptions` gains `listProjects: ListProjects` alongside the
     renamed `registerProject: RegisterProject` (rename from
     `projectService`).
   - Existing `POST /api/projects` handler calls `opts.registerProject.run(name)`
     instead of `opts.projectService.registerProject(name)`.
   - New `GET /api/projects` handler: calls `opts.listProjects.run()`,
     returns `200` with the result as-is — `{ id, name }[]` (empty array if
     none — no special-casing needed).

5. **`api/src/app.ts`**
   - Import `RegisterProject` and `ListProjects` from `./use-cases/...`
     instead of `ProjectService` from `./services/...`.
   - Instantiate both with `projectRepository`, pass both into
     `app.register(projectRoutes, { registerProject, listProjects })`.

6. **`api/src/services/`** — deleted (now empty).

7. **`api/STANDARDS.md`** — replace the "Expected outcomes as typed results"
   bullet's mention of services, and the three-layer bullet, to describe
   `routes → use-cases → repositories`, one class per use case, `run()` as
   the single public method.

8. **`api/CLAUDE.md`** — update the "Architecture" section: `src/services/`
   → `src/use-cases/`, "Services" bullet rewritten for the use-case-class
   pattern.

9. **`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md`** —
   API contract line becomes `GET /api/projects → 200 Array<{ id: string,
   name: string }>`, and remove "Results are ordered by registration date,
   most recent first."

10. **Tests — one file per endpoint:**
    - `api/tests/acceptance/projects.test.ts` — existing `POST` tests,
      unchanged (no direct reference to the renamed class, only
      `app.inject`).
    - `api/tests/acceptance/listProjects.test.ts` — new, covers
      `GET /api/projects`:
      - returns `200 []` when no projects registered
      - returns `200` with all registered projects as `{ id, name }`
        (order-agnostic assertion — sort both sides or use
        `expect.arrayContaining`)

## Not in scope

- Frontend (selector component, sorting, loading/error/empty states) — a
  separate US-1.2 frontend slice.
- Any DB migration — no new column, no ordering.

## Verification

- `pnpm --filter api ci:checks` (lint + typecheck + test) must pass.
- Manually: `curl http://localhost:<port>/api/projects` after registering a
  couple of projects via `POST /api/projects`, confirm `200` with
  `[{ "id": "...", "name": "..." }, ...]`, and `[]` on a fresh DB.

## Execution notes

Single cohesive change (repository → use-cases → route → wiring → docs →
tests, all interdependent) — one implementer, no parallel subtask split.

**TDD, red-green-refactor, per slice** (repository → use-case → route):

1. `ProjectRepository.findAll()` — failing test first (expects `[]` / rows),
   simplest implementation, green, refactor.
2. `ListProjects.run()` — failing behavior via the acceptance test path
   (this repo doesn't unit-test use-cases in isolation currently, see
   existing `POST` tests — stay consistent), simplest implementation, green,
   refactor.
3. `GET /api/projects` route — failing acceptance test (empty list, then
   populated list), simplest implementation, green, refactor.

Rename/move of `ProjectService` → `RegisterProject` is a refactor step, not
new behavior — done under the safety of the existing passing `POST` tests
(no new test needed for the rename itself).
