# US-1.2 — Project Listing: Frontend (`DashboardPage`) — Implementation Plan

**Status**: Approved, not yet implemented.

## Context

`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md` needs the frontend slice
of project listing. The `GET /api/projects` endpoint already exists (see
`docs/_plans/2026-07-11-us-1.2-project-listing-backend.md`). This plan covers the
selector UI, wired to that endpoint.

The current walking-skeleton page (`HomePage`) is repurposed and renamed: it's no
longer a generic landing page, it's the project dashboard. Its existing health-check
content (`fetchHealth`/`HealthState`) is dropped entirely — served its walking-skeleton
purpose, no longer needed once real page content exists.

## Test strategy (per `iteration-specification`)

- **Granularity**: page-level only, per this repo's default (root `STANDARDS.md`,
  "Coarse-grained tests over fine-grained ones") and `web/STANDARDS.md`'s existing
  convention — no dedicated tests for the service/hook/component layers below the page.
- **Typology**: state assertion per rendered UI (loading/success/empty/error), plus one
  interaction case (selecting an option).
- **Stubbed dependencies**: MSW at the network boundary (`GET /api/projects`) only —
  nothing else mocked. `MemoryRouter` wrapper kept for consistency with the existing
  page-test convention.
- **TDD approach**: outside-in, one `DashboardPage.test.tsx` case at a time (see
  Acceptance Criteria below) — write it failing, build only the service/hook/component
  pieces that case needs, green, refactor, move to the next case.

## Acceptance criteria (BDD)

Canonical copy lives in
`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md#acceptance-criteria`;
reproduced here for the implementer's convenience:

- Given the API hasn't responded yet, when `DashboardPage` renders, then a loading
  indicator is shown.
- Given `GET /api/projects` resolves with two projects, when `DashboardPage` renders,
  then both project names appear as dropdown options.
- Given two projects are loaded, when the user selects one via the dropdown, then that
  project becomes the selected value.
- Given `GET /api/projects` resolves `[]`, when `DashboardPage` renders, then an
  empty-state message is shown and no dropdown renders.
- Given `GET /api/projects` fails, when `DashboardPage` renders, then an error message
  is shown.

## Components involved

- **Service** — data fetching only: calls `GET /api/projects`, returns the parsed
  project list or throws.
- **Generic async-state hook** — not project-specific: wraps any fetch function in
  loading/success/error state (cancel-guarded, same pattern the old health-check used),
  reusable for future fetches elsewhere in the app.
- **Selector component** — pure UI: given a project list and a selection callback,
  renders a dropdown. No fetching of its own.
- **Page** (`DashboardPage`, renamed from `HomePage`) — wires the hook + service
  together, holds the selected project id in React state, renders the right thing per
  state.
- **Routing** — the app's root route points at the renamed page.

## Not in scope

- The activity view consuming the selected `project_id` (US-3.1).
- Browser persistence of the selection (resets on refresh, per the user story's Scope).
- Filtering/searching/editing/deleting projects.

## Execution notes

Single cohesive change (service → hook → component → page → routing all
interdependent) — one implementer, no parallel subtask split, worktree isolation.

Build bottom-up in implementation order, but drive it outside-in in test order: start
from the first BDD case's page-level test, build only what that case needs, then move
to the next case — by the last case, all pieces exist, each justified by a case that
needed it.

## Verification

- `pnpm --filter web ci:checks` (lint + typecheck + test) must pass.
- Manually: run the dev server, confirm the dropdown lists registered projects,
  selecting one updates the visible selection, and the empty/error states render
  correctly with no projects registered / the API down.
