# US-1.2 — Project Listing: Frontend (`DashboardPage`) — Implementation Plan

**Status**: Approved, not yet implemented.

## Context

`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md` needs the frontend slice
of project listing. The `GET /api/projects` endpoint already exists (see
`docs/_plans/2026-07-11-us-1.2-project-listing-backend.md`). This plan covers the
selector UI, wired to that endpoint.

The current walking-skeleton page (`HomePage`) is repurposed and renamed: it's no
longer a generic landing page, it's the project dashboard. Its existing health-check
content (`fetchHealth`/`HealthState`) is dropped entirely — served its walking-skeleton
purpose, no longer needed once real page content exists.

## Test strategy (per `iteration-specification`)

- **Granularity**: page-level only, per this repo's default (root `STANDARDS.md`,
  "Coarse-grained tests over fine-grained ones") and `web/STANDARDS.md`'s existing
  convention — no dedicated tests for the service/hook/component layers below the page.
- **Typology**: state assertion per rendered UI (loading/success/empty/error), plus one
  interaction case (selecting an option).
- **Stubbed dependencies**: MSW at the network boundary (`GET /api/projects`) only —
  nothing else mocked. `MemoryRouter` wrapper kept for consistency with the existing
  page-test convention.
- **TDD approach**: outside-in, one `DashboardPage.test.tsx` case at a time (see
  Acceptance Criteria below) — write it failing, build only the service/hook/component
  pieces that case needs, green, refactor, move to the next case.

## Acceptance criteria (BDD)

Canonical copy lives in
`docs/spec/user-stories/2026-06-21-us-1.2-project-listing.md#acceptance-criteria`;
reproduced here for the implementer's convenience:

- Given the API hasn't responded yet, when `DashboardPage` renders, then a loading
  indicator is shown.
- Given `GET /api/projects` resolves with two projects, when `DashboardPage` renders,
  then both project names appear as dropdown options, and a placeholder ("Select a
  project…") is shown as the selected value.
- Given two projects are loaded, when the user selects one via the dropdown, then that
  project becomes the selected value.
- Given `GET /api/projects` resolves `[]`, when `DashboardPage` renders, then an
  empty-state message is shown and no dropdown renders.
- Given `GET /api/projects` fails, when `DashboardPage` renders, then an error message
  is shown.

## Styling

Tailwind CSS v4 via `@tailwindcss/vite` (no PostCSS config, no `tailwind.config.js` —
v4's Vite plugin + a single `@import "tailwindcss";` in `src/index.css`, imported from
`main.tsx`). First styling introduced in this repo — no prior CSS/framework exists.

Per [ADR-008](../architecture/decisions/ADR-008-frontend-architecture.md) (atomic
design), styling is encapsulated at the atom level for reuse, not inlined ad hoc in the
page:

- **`LoadingSpinner`** atom — loading state.
- **`Message`** atom — generic, `variant: 'info' | 'warning' | 'error'` prop mapping to
  Tailwind color classes; only `info` (empty state) and `error` (fetch failure) used by
  this story, `warning`/`success` scaffolded for free for later stories.
- **`ProjectSelector`** atom — native `<select>`, not a custom listbox. Trade-off
  accepted: the OS-rendered popup panel can't be forced dark cross-browser (breaks the
  dark-theme immersion when opened), in exchange for free accessibility/keyboard nav
  and far less code than a hand-built ARIA listbox. First `<option>` is a disabled
  placeholder ("Select a project…") so nothing is auto-selected on load.

Page shell (container max-width, horizontal padding, vertical spacing between
sections) lives in a `PageLayout` component applied once as a react-router **layout
route** (`<Route element={<PageLayout />}>` wrapping child `<Route>`s, `PageLayout`
renders `<Outlet />`) in `App.tsx` — not imported per-page. Every route (including
`NotFoundPage`, and US-3.1 later) gets the shell automatically.

Text sizes and colors: no custom design tokens — use Tailwind's default scale
consistently rather than arbitrary values. Convention to be noted in `web/STANDARDS.md`.

**Look**: Supabase-inspired dark theme (dark mode only for MVP, no toggle).

- Page background `zinc-950`, card/surface `zinc-900`, borders `zinc-800` (1px,
  `rounded-lg`) — borders do the separation work instead of shadows, which barely read
  on dark backgrounds.
- Text: `zinc-100`/`zinc-50` primary, `zinc-400` secondary/muted.
- Accent: `emerald-500`/`emerald-400` (focus rings, active/selected states).
- `Message` variants (lighter shades so they read on dark bg): `info` → `zinc-400` on
  `zinc-800`, `warning` → `amber-400`, `error` → `red-400`.

## Components involved

- **Service** — data fetching only: calls `GET /api/projects`, returns the parsed
  project list or throws.
- **Generic async-state hook** — not project-specific: wraps any fetch function in
  loading/success/error state (cancel-guarded, same pattern the old health-check used),
  reusable for future fetches elsewhere in the app.
- **`ProjectSelector` atom** — pure UI: given a project list and a selection callback,
  renders a dropdown. No fetching of its own.
- **`LoadingSpinner` / `Message` atoms** — see Styling above.
- **`PageLayout`** — page shell, see Styling above.
- **Page** (`DashboardPage`, renamed from `HomePage`) — wires the hook + service
  together, holds the selected project id in React state, renders the right atom/
  component per state.
- **Routing** — `App.tsx` root route points at the renamed page; all routes nested
  under the `PageLayout` layout route.

## Not in scope

- The activity view consuming the selected `project_id` (US-3.1).
- Browser persistence of the selection (resets on refresh, per the user story's Scope).
- Filtering/searching/editing/deleting projects.

## Execution notes

Single cohesive change (service → hook → component → page → routing all
interdependent) — one implementer, no parallel subtask split, worktree isolation.

Build top-down for the page's internal wiring, driven outside-in in test order: start
from the first BDD case's page-level test, implement the fetch + state directly inline
in `DashboardPage` to go green, then extract into a separate service/hook only at the
refactor step once that split is actually justified (duplication, or a later case
needing the same shape) — don't pre-build the service/hook split before a test demands
it. The `LoadingSpinner`/`Message`/`ProjectSelector`/`PageLayout` atoms are the
exception: build these as their own files from the start, not via extraction — they're
a standing architectural decision (ADR-008 atomic design), not speculative structure.

- Manually: run the dev server, confirm the dropdown lists registered projects,
