# US-2.2 — Event Submission

**Slice**: 3 — Core Pipeline  
**Status**: Ready for planning

## User Story

As the telemetry system, skill invocation events from the just-closed session transcript are
submitted to the server, deduplicated by `(session_id, tool_use_id)`.

## Context

Event submission is Phase 2 of `ripe sync`, running after skill registration (Phase 1)
has resolved all IDs. It parses the `.jsonl` session transcript, extracts `Skill` tool invocations,
and POSTs them in a single batch (ADR-001).

Deduplication ensures that re-submitting the same transcript (e.g. after a crash) never inflates
counts. Developer identity is `git config user.email` (ADR-002).

## Scope

**In scope:**

- `POST /api/events` server endpoint
- Phase 2 of `ripe sync`: parse transcript, resolve IDs, submit batch
- Server-side deduplication on `(session_id, tool_use_id)`
- `ripe init` adds `SessionEnd` hook to `.claude/settings.local.json` if not present

**Out of scope:**

- Non-Skill tool invocations (ignored)
- Retry logic on network failure (log and exit 0)

## API Contract

```text
POST /api/events
Body: Array<{
  skill_id: string      // from local cache (server-assigned)
  skill_name: string    // current name at time of submission
  timestamp: string     // ISO 8601
  user_email: string    // git config user.email
  project_id: string    // from local cache
  session_id: string    // Claude Code session ID
  tool_use_id: string   // unique per invocation within the session
}>
→ 204 No Content
```

Server behavior: bulk insert with `ON CONFLICT(session_id, tool_use_id) DO NOTHING`.

## Client Behavior (Phase 2 of `sync`)

1. Parses the `.jsonl` transcript for `Skill` tool use entries
2. Skips non-Skill tool uses
3. Skips namespaced skills (containing `:`) with a stderr warning
4. Resolves `skill_id` from the local cache (populated in Phase 1)
5. Sends all events in one POST
6. If no skill invocations found: exits 0, no request made
7. If `git config user.email` is unset: stderr warning, event sent with empty string
8. If server unreachable: stderr warning, exits 0

## Acceptance Criteria

- [ ] All skill invocations in the transcript are submitted in one request
- [ ] Duplicate submissions (same `session_id + tool_use_id`) are ignored by the server
- [ ] No request made if the transcript contains no skill invocations
- [ ] Non-Skill tool uses are ignored
- [ ] Missing `user_email` → warning on stderr, event submitted with empty string
- [ ] Server unreachable → stderr warning, exit 0 (does not block session close)
