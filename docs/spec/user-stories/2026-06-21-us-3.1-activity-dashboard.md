# US-3.1 — Activity Dashboard

**Slice**: 4 — Activity Feature  
**Status**: Ready for planning

## User Story

As a team member, I can view a ranked list of skill invocations for the selected project, filtered
by date range, so I know which skills are being used.

## Context

This completes the full vertical slice: events are ingested (US-2.2), and now they are visible.
The project selector from US-1.2 provides the `project_id`. The activity endpoint JOINs `skills`
and `events` server-side, so skill renames are handled transparently (ADR-007).

Data fetching strategy: fetch on every filter change, no client-side cache (ADR-008).  
Frontend architecture: atomic design — atoms, molecules, organisms, pages (ADR-011).

## Scope

**In scope:**

- `GET /api/projects/{project_id}/activity` server endpoint with date range filtering
- `FiltersPanel` component (two date inputs)
- `SkillsTable` component (ranked list)
- Loading, error, and empty states

**Out of scope:**

- Per-developer breakdown (deferred to v2 per epic-1-scope.md)
- Export or download of data
- Charts or visualisations beyond a ranked table

## API Contract

```
GET /api/projects/{project_id}/activity?from_date=YYYY-MM-DD&to_date=YYYY-MM-DD
→ 200 {
  skills: Array<{
    skill_id: string
    name: string        // current name — server-side JOIN with skills table
    invocations: number
  }>
}
```

- Results sorted by `invocations` descending
- `from_date` and `to_date` are inclusive
- `name` reflects the skill's current name; renames do not reset the count
- Empty `skills` array if no matching events

## Frontend Behavior

- `FiltersPanel`: date range inputs; on change, refetches from server (no client-side filtering)
- Default date range: last 30 days
- `SkillsTable`: renders ranked rows (rank, name, invocation count)
- Spinner shown during fetch; previous results preserved until new data arrives (no flicker)
- Error state shown if server is unreachable
- Empty state shown if no invocations in the selected range

## Acceptance Criteria

- [ ] Skills ranked by invocation count, descending
- [ ] Date filter limits which events are counted (server-side)
- [ ] Renamed skills show current name; count is continuous across the rename
- [ ] Changing the date filter refetches from server
- [ ] Spinner shown during fetch; previous results not cleared until new data arrives
- [ ] Empty state shown if no invocations in range
- [ ] Error state shown if the request fails
