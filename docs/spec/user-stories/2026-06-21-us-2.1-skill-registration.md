# US-2.1 — Skill Registration

**Slice**: 3 — Core Pipeline  
**Status**: Ready for planning

As the telemetry system, committed skills in `.claude/skills/` are registered with the server,
with their assigned IDs cached in a committed file and shared across the team.

## Context

Skill registration happens in two places:

1. **`ripe init`**: registers all committed skills from main branch (initial setup)
2. **`ripe sync` Phase 1**: registers new committed skills not yet on server (ongoing)

Only skills **committed on main branch** are tracked — uncommitted files and skills only on
feature branches are ignored, even if invoked. Skill IDs are cached in `.ripe/skills.json`
(committed to repo) for team consistency. Server is the source of truth for IDs (ADR-015).
Skills are project-scoped: `(project_id, name)` uniquely identifies a skill.

**Rename handling:** V2 feature. In V1, renaming = delete + add (orphans historical data).

## Scope

**In scope:**

- `POST /api/skills` server endpoint
- `ripe init`: register committed skills from main branch
- Phase 1 of `ripe sync`: register new committed skills
- Committed cache file `.ripe/skills.json` with `(name → skill_id)` mappings
- Skip uncommitted and namespaced skills

**Out of scope:**

- Skill deletion or archival
- Rename detection (V2)
- Skills from directories other than `.claude/skills/`
- Tracking uncommitted/WIP skills

## API Contract

```text
POST /api/skills
Body: {
  project_id: string,
  skills: Array<{ name: string }>
}
→ 200 {
  skills: Array<{ name: string, skill_id: string }>
}
```

Server behavior: `ON CONFLICT(project_id, name) DO NOTHING` — same `(project_id, name)` always
returns the same `skill_id`. Only skills absent from the local cache are sent.

## Client Behavior

**`ripe init`:**

0. If `.ripe/config.json` does not exist: exit 0 silently (project not initialized via `ripe init`)
1. Scans `.claude/skills/` on main branch for committed skills
2. Skips namespaced skills (containing `:`) — stderr warning
3. POSTs all skills to `POST /api/skills`
4. Writes returned `skill_id`s to `.ripe/skills.json` (committed file)
5. Server unreachable → stderr error, exit 1 (blocks init)

**Phase 1 of `ripe sync`:**

1. Scans `.claude/skills/` for skills committed on main branch only
2. Skips uncommitted files and skills only on feature branches — stderr warning: "skill X not on main, skipping"
3. Skips namespaced skills (containing `:`) — stderr warning
4. For skills absent from `.ripe/skills.json`, POSTs to `POST /api/skills`
5. Writes returned `skill_id`s to `.ripe/skills.json`
6. Server unreachable → stderr warning, exit 0 (non-blocking)

## Acceptance Criteria

- [ ] `ripe init` registers all skills committed on main branch
- [ ] `ripe sync` registers new skills committed on main, not in `.ripe/skills.json`
- [ ] Uncommitted skills and skills only on feature branches are skipped with stderr warning during `sync`
- [ ] Skill IDs are written to `.ripe/skills.json` (committed file)
- [ ] Already-cached skills are not included in POST payload
- [ ] Same `(project_id, name)` always returns same `skill_id` (server idempotency)
- [ ] Namespaced skills (containing `:`) are skipped with stderr warning
- [ ] Server unreachable during `init` → stderr error, exit 1
- [ ] Server unreachable during `sync` → stderr warning, exit 0
- [ ] Empty `.claude/skills/` → no POST made
