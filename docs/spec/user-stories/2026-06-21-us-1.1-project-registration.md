# US-1.1 — Walking Skeleton & Project Registration

**Slice**: 1 — Foundation & Project Registration  
**Status**: Done

## User Story

As a developer, I can install `ripe`, run `ripe init <server-url>` once per project, and have the
full deployment pipeline proven end-to-end, so the system is ready to receive data.

## Context

This is the first deliverable. It combines three concerns: (1) proving end-to-end deployment
connectivity via a walking skeleton, (2) publishing the `ripe` CLI to GitHub Packages, and
(3) registering a project with the API server. No business logic beyond project registration lands here —
everything else builds on this slice.

The `project_id` is server-assigned — clients do not generate it (ADR-015).  
Developer identity uses `git config user.email` (ADR-002).  
Global hook + per-project opt-in via `init` (ADR-003).  
Tech stack (ADR-004): TypeScript + Fastify (server), pnpm workspace.  
Deployment (ADR-013): Railway for hosting, GitHub Actions for CI/CD.  
Environments (ADR-013): staging + production as separate Railway services.

## Scope

**In scope:**

- pnpm workspace bootstrap (`pnpm-workspace.yaml`, root `package.json` with shared scripts)
- Fastify API server skeleton with `GET /api/health` and `POST /api/projects` endpoints
- Dockerfile for production build
- Railway configuration for staging and production services
- `.env.example` documenting all required variable names (no values); `.env` gitignored
- GitHub Actions CI/CD: lint, type checks, and tests on push; auto-deploy to staging on merge to
  `main`; production deploy via manual trigger
- `ripe` CLI package published to GitHub Packages as `@paulelian-tabarant/ripe`
- `ripe init <server-url>` CLI command
- Local committed config storing `project_id` and `server_url`

**Out of scope:**

- Web dashboard (introduced in Slice 2)
- Database tables beyond `projects` (skills and events come in Slice 3)
- Authentication (MVP has none — ADR-004 / version-1-scope.md)

## API Contracts

```text
GET /api/health
→ 200 { "status": "ok" }

POST /api/projects
Body: { name: string }   ← root directory name of the project
→ 201 { "projectId": "proj_<nanoid>" }          (new project)
→ 409 { "projectId": "proj_<nanoid>", "message": "Project already exists" }  (duplicate name)
→ 400                                            (invalid body)
```

`name` is read from the current working directory name. Names are UNIQUE on the server — a
duplicate triggers a 409 (see ADR-017). The CLI prompts the user to confirm before reusing an
existing `projectId`. `projectId` format: `proj_` prefix + nanoid (~26 chars, URL-safe).

## Deployment Strategy

- **On push (any branch)**: run lint, type checks, and tests
- **On merge to `main`**: auto-deploy to staging; staging database resets on each deploy
- **Production**: manual trigger from the Railway dashboard after staging validation
- **Main branch**: protected — changes land via pull requests only

## CLI Behavior

1. Reads the project name from the current working directory
2. If `.ripe/config.json` already exists: warns user and exits 0 (avoids creating a duplicate entry)
3. POSTs `{ name }` to `<server-url>/api/projects`
4. On 201: persists `projectId` and `serverUrl` to `.ripe/config.json` (committed to git), exits 0
5. On 409: prompts `Project '<name>' already exists. Do you want to use it? (y/n)` — on `y` writes
   the returned `projectId` to config and exits 0; on `n` exits 0 without writing
6. On unreachable server or other error: prints to stderr, exits 1

## Acceptance Criteria

- [ ] `GET /api/health` returns `{ "status": "ok" }`
- [ ] `ripe init <server-url>` creates `.ripe/config.json` with `projectId` and `serverUrl`
- [ ] Project name is derived from the current directory name
- [ ] Running `init` when config already exists warns and exits 0
- [ ] `POST /api/projects` with a duplicate name returns 409 with the existing `projectId`
- [ ] CLI prompts on 409 — `y` writes config, `n` exits 0 without writing
- [ ] API server unreachable → stderr error, exit 1
- [ ] `.env.example` is present and documents all required variable names
- [ ] CI runs lint, type checks, and tests on every push; failing CI blocks the deploy
- [ ] Merging to `main` automatically deploys to staging
- [ ] Production deployment requires a manual trigger
- [ ] Deployed API server responds to `GET /api/health` at both staging and production URLs
- [ ] CLI package published to GitHub Packages as `@paulelian-tabarant/ripe`
